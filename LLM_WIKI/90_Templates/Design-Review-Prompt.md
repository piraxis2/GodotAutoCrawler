# Design Review Prompt

아래 프롬프트는 Task나 ADR을 구현하기 전에 설계의 완결성, 범위, 위험을 검토할 때 사용한다.

```text
AGENTS.md와 LLM_WIKI를 먼저 읽고 실제 코드와 대조해라.

이번 세션은 설계 리뷰 전용이다. 제품 코드와 리소스는 수정하지 마라.

검토 대상:
- Task: [Task 문서 경로]
- ADR: [ADR 문서 경로 또는 없음]
- 관련 시스템: [System 문서 경로]

검토 목표:
- 요구사항을 구현 가능한 독립 Step으로 분해했는지 확인한다.
- 현재 코드 구조와 충돌하거나 중복되는 책임을 찾는다.
- public API, 데이터 모델, 수명 주기, 오류 정책, 저장 호환성을 검토한다.
- 완료 조건이 관찰 가능하고 테스트 가능한지 확인한다.
- 구현 전에 결정해야 할 미정 사항을 식별한다.

필수 검토 항목:
1. 목표와 Scope/Out of Scope가 명확한가?
2. 각 Step의 선행 조건과 완료 조건이 명확한가?
3. 기존 시스템과 책임이 중복되지 않는가?
4. 데이터 타입, 기본값, 저장/로드, 버전 정책이 충분한가?
5. signal, 재진입, 반복 실행, 객체 수명 주기 위험이 있는가?
6. 테스트 대역과 자동 검증이 가능한 구조인가?
7. 향후 확장 때문에 지금 과도하게 일반화하고 있지 않은가?

결과 형식:

## Findings
- P0~P3 심각도순 설계 문제
- 관련 Task/ADR 절과 실제 코드 위치
- 왜 구현 전에 해결해야 하는지
- 권장 수정 방향

## Open Decisions
- 구현 전에 사용자가 결정해야 할 사항
- 권장 선택과 trade-off

## Step Assessment
- 각 Step의 입력, 출력, 선행 조건, 완료 조건 평가
- 합치거나 나눠야 할 Step

## Verification Assessment
- 필요한 단위/통합/회귀 테스트
- 현재 계획에서 빠진 실패 시나리오

## Verdict
- Approved
- Approved after design fixes
- Rework required

설계에 문제가 없으면 없다고 명확히 말하고, 남아 있는 가정과 테스트 공백을 적어라.
코드 구현은 시작하지 마라.

필요한 경우에만 Task/ADR/Wiki 문서 수정안을 제안하거나 문서만 수정한다.
```

## Usage Notes

- 설계 리뷰에서는 제품 코드, `.tscn`, `.tres`를 수정하지 않는다.
- Task/ADR 문서 수정도 사용자가 요청했거나 리뷰 결과 반영이 명확할 때만 한다.
- `Approved` 판정 전에는 구현 프롬프트로 넘어가지 않는다.
