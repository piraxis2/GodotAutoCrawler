# Step -> Review 개발 지침

이 문서는 큰 작업을 작은 Step으로 나누어 구현하고, 각 Step을 독립적으로 리뷰·수정·검증하는 개발 절차를 정의한다.

목표는 빠르게 많은 코드를 작성하는 것이 아니라, 매 단계마다 동작하는 기준점을 만들고 설계 오류가 다음 단계로 전파되지 않게 하는 것이다.

## 기본 원칙

1. 한 번에 하나의 Step만 구현한다.
2. 각 Step은 독립적으로 설명하고 검증할 수 있어야 한다.
3. 구현 담당자는 기존 사용자 변경을 되돌리지 않는다.
4. 리뷰는 기능 요약보다 버그, 회귀, 데이터 손실, 설계 경계 위반을 먼저 찾는다.
5. 리뷰 지적을 수정한 뒤 반드시 재검증한다.
6. 치명적인 문제가 남아 있으면 다음 Step으로 넘어가지 않는다.
7. 구조 개선과 기능 추가는 가능한 한 같은 Step에 섞지 않는다.

## 역할

### 구현 담당

- Step의 목표와 완료 조건을 먼저 확인한다.
- 관련 코드를 읽고 기존 패턴을 따른다.
- 변경 범위를 Step 안으로 제한한다.
- 구현 후 테스트와 정적 검토를 수행한다.
- 변경 내용, 검증 결과, 남은 위험을 보고한다.

### 리뷰 담당

- 요구사항 대비 실제 코드를 확인한다.
- 구현 보고만 믿지 않고 실행 경로와 데이터 왕복을 추적한다.
- 문제를 심각도순으로 제시한다.
- 가능하면 파일과 라인 위치를 함께 기록한다.
- 수정 후 완료 여부를 다시 판정한다.

구현과 리뷰는 서로 다른 사람이 맡는 것이 좋지만, 동일한 담당자가 별도 리뷰 단계로 수행해도 된다.

## 전체 흐름

```text
작업 분해
  -> Step 목표 및 완료 조건 작성
  -> 구현
  -> 구현 결과 보고
  -> 코드 리뷰
  -> 지적 사항 수정
  -> 재검증
  -> Step 완료 판정
  -> 다음 Step
```

## 1. Step 정의

각 Step은 아래 형식으로 작성한다.

```markdown
## Step N: 이름

목표:
- 이 단계에서 사용자가 얻는 결과

작업 범위:
- 수정할 시스템과 주요 파일
- 이번 단계에서 구현할 동작

제외 범위:
- 다음 단계로 미룰 기능
- 의도적으로 건드리지 않을 구조

완료 조건:
- 관찰하거나 테스트할 수 있는 결과
- 오류 처리 및 호환성 기준

검증 방법:
- 자동 테스트
- Godot headless/editor 실행
- 샘플 리소스 왕복 저장
- 수동 실행 시나리오
```

Step은 보통 한 가지 사용자 흐름이나 하나의 구조적 경계를 다루도록 한다. 완료 조건이 여러 문단으로 길어진다면 Step을 더 나누는 편이 좋다.

## 2. 구현 보고

구현 담당자는 완료 후 아래 형식으로 보고한다.

```markdown
## 구현 결과

변경 파일:
- `path/to/file.gd`: 변경 목적

구현 내용:
- 실제로 달라진 동작
- 주요 설계 판단

검증:
- 실행한 명령 또는 테스트
- 성공/실패 결과

남은 위험:
- 아직 검증하지 못한 경로
- 의도적으로 다음 Step으로 미룬 항목
```

“구현 완료”만 보고하지 않는다. 어떤 조건으로 완료되었다고 판단했는지 반드시 포함한다.

## 3. 리뷰 기준

리뷰는 다음 순서로 확인한다.

### 기능 정확성

- Step의 완료 조건을 실제 코드가 만족하는가?
- 정상 흐름과 종료 흐름이 모두 연결되어 있는가?
- 빈 값, 잘못된 연결, 누락된 리소스를 안전하게 처리하는가?

### 데이터 안정성

- 저장 후 다시 로드했을 때 값이 보존되는가?
- UI -> Resource -> Runtime 변환에서 순서와 타입이 유지되는가?
- 이전 버전 리소스와의 호환성이 필요한가?
- 편집만 하고 저장했을 때 기존 데이터가 사라지지 않는가?

### 상태와 수명 주기

- signal이 연결되기 전에 발생하지 않는가?
- `queue_free()` 전후의 지연 signal이 새 객체를 건드리지 않는가?
- 재진입, 연속 실행, 중간 교체가 안전한가?
- 같은 UI를 여러 번 열었을 때 항목이나 signal이 누적되지 않는가?

### 설계 경계

- 에디터 코드가 런타임 경로에 들어오지 않는가?
- 데이터 모델이 UI 위젯을 직접 알지 않는가?
- 이번 Step이 불필요하게 다른 시스템까지 확장되지 않았는가?
- 새 추상화가 실제로 사용되고 있는가?

### 검증 가능성

- 핵심 경로를 재현할 샘플이나 테스트가 있는가?
- Godot 스크립트가 파싱되는가?
- 구현 보고의 테스트 결과를 재현할 수 있는가?

## 4. 심각도

리뷰 지적은 다음 기준을 사용한다.

- **P0**: 데이터 파괴, 보안 문제, 프로젝트 실행 불가. 즉시 수정하며 다음 Step 진행 금지.
- **P1**: 핵심 기능 실패, 저장 데이터 손실, 일반 사용 흐름의 크래시. Step 완료 전에 수정.
- **P2**: 특정 조건의 오동작, 구조적 부채, 유지보수 위험. 가능하면 Step 내 수정하고, 미룰 경우 후속 Step을 명시.
- **P3**: 작은 결함, 명명, 중복, 제한적인 UX 문제. 별도 정리 가능.

리뷰 결과는 발견 사항을 먼저 제시한다.

```markdown
## 발견 사항

1. **[P1] 문제 제목**
   - 발생 조건
   - 사용자 영향
   - 파일/라인
   - 권장 수정 방향

## 검증 결과
- 통과한 항목
- 실행하지 못한 항목

## 판정
- 완료 / 수정 후 완료 / 미완료
```

## 5. 수정 결과 보고

리뷰 지적을 처리한 뒤에는 각 항목을 대응시켜 보고한다.

```markdown
## 처리 결과

### [P1] 문제 제목 - 수정
- 변경 내용
- 회귀 방지 방법
- 검증 결과

### [P2] 문제 제목 - 보류
- 보류 이유
- 영향 범위
- 처리할 후속 Step
```

보류는 “시간이 없어서”가 아니라 범위, 위험, 의존성 관점에서 설명한다. P0/P1은 원칙적으로 보류하지 않는다.

## 6. 재검증과 완료 판정

리뷰 담당자는 수정된 코드에서 다음을 확인한다.

1. 지적한 문제가 실제로 제거됐는가?
2. 수정 때문에 새로운 회귀가 생기지 않았는가?
3. 자동 테스트 또는 재현 시나리오가 통과하는가?
4. Godot headless/editor 로드가 성공하는가?
5. 다음 Step이 현재 Step의 미완성 부분에 의존하지 않는가?

판정은 다음 중 하나로 한다.

- **완료**: 완료 조건 충족, P0/P1 없음, 후속 작업이 명확함.
- **수정 후 완료**: 작은 P2/P3만 남고 다음 Step을 막지 않음.
- **미완료**: 핵심 흐름 실패, 데이터 손실 위험 또는 검증 불가.

점수는 의사소통 보조 수단일 뿐 완료 조건을 대신하지 않는다.

## 7. Godot 프로젝트 권장 검증

DialogueTool과 같은 Godot 에디터 플러그인은 최소한 다음을 확인한다.

```text
1. Godot headless editor 로드
2. 플러그인 활성화 및 GraphEdit 표시
3. 노드 생성 -> 값 입력 -> 저장
4. 리소스 재로드 -> 값 보존 확인
5. 런타임 실행 -> 종료까지 진행
6. 같은 리소스 반복 실행
7. 실행 중 다른 리소스로 교체
8. 에디터 디버그 하이라이트 종료 후 복원
```

가능한 경우 다음과 같은 작은 통합 그래프를 회귀 기준으로 유지한다.

```text
Start
  -> Say
  -> Choice
  -> Branch(Expression 또는 Variable)
  -> Say A / Say B
  -> End
```

## 8. 프롬프트 단계 분리

Agent 작업은 다음 세션 역할을 섞지 않는다.

```text
설계 작성
  -> 설계 리뷰
  -> 설계 승인
  -> Step 구현
  -> 코드 리뷰
  -> 수정 및 재검증
```

### 설계 리뷰

- Task와 ADR의 구현 가능성, 범위, API, 데이터 계약, 테스트 계획을 검토한다.
- 제품 코드는 수정하지 않는다.
- 결과는 `Approved`, `Approved after design fixes`, `Rework required`로 판정한다.
- 템플릿: [[Design-Review-Prompt]]

### 구현

- 승인된 설계의 Step 하나만 구현한다.
- 구현 중 설계 변경이 필요하면 임의로 확장하지 않고 Design Deviation을 보고한다.
- 템플릿: [[Implementation-Prompt]]

### 코드 리뷰

- 구현 결과가 승인된 Task/ADR과 Completion Criteria를 만족하는지 검토한다.
- 버그, 회귀, 데이터 손실, 수명 주기 문제를 P0~P3로 분류한다.

## 9. 간단한 구현 프롬프트

```text
현재 프로젝트에서 Step N만 진행해라.

목표:
[목표]

작업 범위:
[범위]

제외 범위:
[이번에 하지 않을 일]

완료 조건:
[관찰 가능한 완료 조건]

진행 규칙:
- 먼저 관련 파일을 읽고 현재 상태를 요약한다.
- 기존 사용자 변경을 되돌리지 않는다.
- 변경 범위를 이 Step에 제한한다.
- 구현 후 테스트 또는 Godot headless 검증을 수행한다.
- 변경 파일, 구현 내용, 검증 결과, 남은 위험을 보고한다.
```

상세한 구현 세션에는 [[Implementation-Prompt]]를 우선 사용한다.

## 10. 코드 리뷰 프롬프트

```text
완료된 Step N을 코드 리뷰해라.

Step 목표와 완료 조건을 기준으로 실제 코드를 확인한다.
구현 보고만 신뢰하지 말고 저장/로드, 런타임 경로, signal 수명 주기와 재진입을 추적한다.

결과 형식:
1. 발견 사항을 P0~P3 순서로 제시
2. 파일과 라인 위치 포함
3. 잘 된 부분
4. 검증한 내용과 검증하지 못한 내용
5. 완료 / 수정 후 완료 / 미완료 판정

문제가 없으면 없다고 명확히 말하고 남은 테스트 공백을 적는다.
```
